엔진에서 Transform을 완전히 이해하기

Posted on 2026-01-23 In 1. Engine Views:

이 정리는 다음 문제를 해결하기 위해 만들어졌다 (with GPT)

  • 수식은 이해하지만 엔진에 적용하면 헷갈린다
  • 로컬 / 월드 회전이 뒤섞인다
  • LookAt, View, Projection이 외워서 쓰는 코드가 된다
  • column-major / row-major 때문에 곱 순서가 헷갈린다

사실 내가 헷갈려서 정리를 했다.

1. 좌표계는 수학이 아니라 엔진의 세계관이다

엔진을 만들 때 가장 먼저 해야 할 일은 좌표계를 정의하는 것이다.

(지금 정의하는 좌표계는 world 좌표계)

반드시 고정해야 할 요소들:

  • 오른손 / 왼손 좌표계 중 무엇을 쓸 것인가
  • Forward 방향은 +Z 인가, -Z 인가
  • Z 값은 커질수록 멀어지는가, 가까워지는가

이걸 명확히 하지 않으면:

  • 회전 부호가 뒤집히고
  • LookAt이 맞지 않고
  • Projection depth가 깨진다

👉 좌표계는 코드보다 먼저 정해져야 한다

2. Transform의 본질: 공간을 바꾸는 일

Transform은 “값을 바꾸는 것”이 아니라

좌표 공간을 변환하는 것이다.

실제 객체가 렌더링 되기까지 많은 변환 과정을 거친다.

그래서 행렬은 항상 이런 의미를 가진다.

  • world matrix : Local → World
  • view matrix : World → Camera
  • projection matrix : Camera → Clip

World Matrix

  • Local Space → World Space
  • 객체의 위치, 회전, 스케일을 표현
  • 객체마다 다르다
  • 위치 / 회전 / 스케일 변경 시 업데이트

View Matrix

  • World Space → Camera Space
  • 카메라의 World Matrix의 역행렬
  • 카메라가 움직이면 View가 바뀐다
  • 입력 처리 이후 업데이트

의미적으로는

“카메라를 움직이는 게 아니라

세상을 카메라 기준으로 반대로 이동시킨다”

Projection Matrix

  • Camera Space → Clip Space
  • FOV, Aspect, Near/Far, Depth Range
  • 렌더링 설정 값
  • 창 크기 변경, 카메라 설정 변경 시 업데이트
  • 매 프레임 업데이트 하지 않는다

👉 논리 업데이트와 분리되어야 한다

이동과 회전

**이동(Translation)**은

  • 기준점이 필요없고
  • 항상 월드 기준이다.

**회전(Rotation)**은

  • 반드시 기준점(pivot)이 필요하다

  • 항상 원점 기준으로 발생한다.

  • 그래서 등장하는 수식이 원점을 이동해서 회전 후 다시 원점으로 돌아가는 로컬 회전 방식이다.

  • 로컬회전 vs 월드 회전

    • 행렬 곱의 순서에 따라 축이 바뀐다.

3. LookAt의 본질

LookAt은 각도를 계산하는 함수가 아니다.

LookAt이 하는 일은:

  • 어떤 방향을 바라보도록
  • 객체(또는 카메라)의 로컬 좌표계를 재구성하는 것

즉 forward / right / up 축을 새로 만드는 과정이다.

LookAt의 순서

  1. 방향 벡터 계산

    이 때, 좌표계의 정의가 영향을 준다.

    (forward가 +z인지, 오른손 좌표계인지 등)

    방향만 필요한 것이니 normalize해준다.

    그리고 좌표계는 회전행렬의 축이 되므로

    길이가 1이고 서로 직교해야한다.

    안하면 회전할 때 스케일이 묘하게 변함,,

  2. right / up 축 생성

    forward를 알고 worldUp을 알면 right을 구할 수 있고,

    forward와 right을 알면 up을 구할 수 있음

    왼손 좌표계라면:

  3. 회전 행렬 구성

    여기서 헷갈리면 100% column / row major 문제다.

    • OpenGL / GLSL → column-major
    • M * v 기준

4. Column-major / Row-major의 올바른 이해

Row-major / Column-major는
메모리 배치 문제이지, 수학 문제는 아니다

major는 설계 선택이 아니라
데이터를 GPU로 전달하는 방식의 문제다.

중요한 건 CPU와 GPU가 같은 의미로 행렬을 해석하느냐이다.

차이는

  • 벡터가 왼쪽에 오느냐 (v * M)
  • 오른쪽에 오느냐 (M * v)

OpenGL/GLSL 기준:

👉 CPU 코드도 같은 의미 순서를 유지해야 한다

  • 원하는 major로 일관되게 쓰면 됨 !
  • 대신 그래픽스 API에서 요구하는 쓰기편한 방식은 있음

5. 각도(angle)는 결과물이다

엔진에서 “각도를 계산한다”는 말의 실제 의미는 이거다.

각도는:

  • 입력값이 아니라
  • 방향을 표현하는 파생 결과

그래서 실무에서는:

  • 방향은 벡터로 다루고
  • 회전 상태는 Quaternion으로 저장한다

왜 Quaternion을 쓰는가

매트릭스 기반 회전의 문제:

  • 로컬 축이 계속 변한다
  • 각도 누적이 깨진다
  • 짐벌락 가능성

Quaternion의 역할:

  • 회전을 “과정”이 아니라 “상태”로 저장
  • 누적에 강함
  • 보간에 유리
  • 순서는

👉 각도를 안 쓰기 위해 Quaternion을 쓴다

🩸잘못된 예시 (“각도를 입력으로 쓰는 설계”의 한계)

이 방식의 문제점은

  • yaw/pitch를 상태로 저장
  • 각도를 계속 누적
  • 어느 순간:
    • 축이 꼬이고
    • pitch가 뒤집히고
    • 짐벌락 발생

💧올바른 방법

  1. 두 점으로 방향벡터 생성

  2. 방향 벡터 → Quaternion (회전 상태 생성)

    이 rotation은 **“**지금 어떤 방향을 보고 있는지” 에 대한 회전 결과 상태이다

  3. Quaternion 상태로 저장

    내부 저장상태는

  4. Quaternion → Matrix

    👉 GPU는 행렬만 필요

    👉 논리 상태는 Quaternion이 관리

    만약 디버그용으로 yaw/pitch 가 필요하다면

    방향 →각도 역산으로 구한다

왜 이 방식이 짐벌락을 피하냐면

각도 방식은 yaw→pitch→roll 순서가 고정되어있는데

Quaternion 방식은 현재 방향 상태 + 회전 변화량 방식이라

축이 아닌 방향을 저장하기 때문이다.
