Platform 설계

Posted on 2026-02-05 In 1. Engine Views:

엔진 제작에 있어 platform 레이어는 단순한 윈도우 창 띄우기가 아니다.

핵심은 누가 죽도권을 갖느냐와 환경이 바뀌어도 엔진의 로직이 변하지 않게 추상화하는가에 있다.

이 글은 **엔진 주도 루프(Engine-Owned Loop)**를 기준으로 작성했다..!
많은 프레임워크가 OS의 메시지 루프에 엔진을 종속시키지만(Event-Driven), 확장성을 위해서는 엔진이 루프의 주도권을 쥐어야 하기 때문이다.

Engine-Owned Loop의 platform :

엔진이 OS/ 하드웨어 / 런타임 환경과 상호작용하기 위한 추상계층이며

엔진이 주도권을 가지고 platform은 결정권이 없어야 한다.

엔진은 플랫폼이 윈도우인지 리눅스인지 서버 환경인지 몰라도 동작할 수 있어야 한다.

GameWindow와 Unity 방식은

  • 플랫폼이 루프를 소유
  • 엔진은 콜백으로 호출됨
  • 입력/렌더/시간이 섞임

이 방식은 빠르게 게임을 만들기엔 편하지만 규모가 커질 수록 통제가 어려워진다.

또한 입력시점이 불명확하고 fixed timestep 구현이 꼬이는 문제점이 있다.

그리고 Headless도 불가능하다.

그렇지만 engine과 platform이 분리된 구조는

  • 엔진이 루프를 소유
  • 플랫폼은 수동적으로 응답
  • 입력/시간/렌더가 분리됨

이 구조는 장기적으로 유지하기에 유리하다.

프레임 경계가 명확하고 입력/시간/렌더가 완전히 분리가 된다.

이렇게 하면 테스트가 가능하고 서버가 가능하다.

그럼 게임 뿐만아니라 다른 시뮬레이션 엔진으로도 활용이 가능하게 된다.

이제 책임을 나누면 구조가 뚜렷해진다.

  • Platform 책임
    • 현재 시각 제공 (GetRawTimeSeconds)
      • Time & Precision: 고정밀 타이머를 통한 프레임 시간 계산
    • 이벤트 수집 (PollEvents)
      • Window Management: 창 생성, 크기 조절, 라이프사이클 관리
      • Input Handling: OS 이벤트를 엔진이 이해할 수 있는 데이터로 정제
    • 화면 제출 (SwapBuffers)
      • Context Bridge: 그래픽 API(Renderer)와 OS 윈도우를 연결
  • Engine 책임
    • deltaTime 계산
    • accumulator 관리
    • fixed timestep 유지
    • interpolation 계산

1. 엔진은 언제 입력(InputState)을 알아야 할까 ?

바로 프레임의 시작 시점이다.

InputState는 한 프레임동안 유효한 입력의 스냅샷이다.

InputState의 최소 구성은

  • 키보드 상태
  • 마우스 버튼 상태
  • 마우스 위치
  • 마우스 이동량
  • 휠 스크롤

Platform 내부에는 반드시 buffer가 존재한다.

Platform이 엔진에게 곧바로 InputState를 만들어서 전달하는 구조는

겉보기에는 단순하지만, 실제로는 프레임 경계를 깨뜨리는 설계다.

PollEvents() 중에 입력을 즉시 처리하면 프레임 중간에 데이터가 변해 로직의 일관성이 깨질 수도 있다.

그래서 이벤트 수집과 상태 스냅샷을 분리해야한다.

그래서 Platform 내부에는 반드시 다음 계층이 존재한다.

InputBuffer는 이름 그대로 버퍼다.

이벤트가 아니고 현재까지 관측된 상태이다.

여기서 핵심은 InputBuffer와 InputState의 분리다.

  • InputBuffer

    • OS 이벤트에 의해 갱신되는 상태 버
    • 프레임 중간에도 계속 변함
    • Platform 내부 전용

  • InputState

    • 프레임 시작 시점에 한 번 생성(single thread)
      • render thread일 경우 입력 지연 최소화를 위해 프레임 시작 전에 poll하고 render 직전에 다시 업데이트 하는 경우도 있다 (특히, fps)
    • 읽기 전용
    • 엔진 전체가 공유

이 구조 덕분에 엔진은

언제 입력을 읽어야 하는지 를 명확히 정의할 수 있다.
엔진은 입력을 프레임의 시작 시점에만 알아야 한다.

2. PollEvents는 정확히 무엇을 하는가

많은 사람들이 PollEvents()를 업데이트 함수처럼 오해한다.

하지만 이 함수의 실제 의미는 훨씬 제한적이다.

이 한 줄에서 일어나는 일은 다음뿐이다.

  • OS 메시지 큐를 펌프한다
  • GLFW/Win32/SDL 콜백이 실행된다
  • 그 콜백들이 InputBuffer를 갱신한다

여기서 절대 InputState를 생성하면 안된다

왜 PollEvents에서 InputState를 만들면 안 되냐면

OS 이벤트는 비동기적이다.

  • 키 입력은 언제 올지 모른다
  • 마우스 이동은 프레임 중간에도 발생한다
  • 창 이벤트는 예측 불가능하다

만약 PollEvents 중간에 InputState를 만든다면:

  • 시스템 A는 이전 입력을 보고
  • 시스템 B는 이후 입력을 보게 된다

이건 버그가 아니라 설계 실패다.

그래서 InputState는 반드시:

프레임 경계(Frame Boundary)에서 단 한 번만 생성되어야 한다.

다시 말해

PollEvents 중에 입력을 즉시 처리하면 프레임 중간에 데이터가 변해 로직의 일관성이 깨질 수 있다.

따라서 **이벤트 수집(Poll)**과 **상태 스냅샷(GetState)**을 분리해야 한다

GetInputState에서

  • InputBuffer의 현재 상태를 복사
  • 이전 프레임 상태와 비교
  • Pressed / Released를 계산
  • immutable InputState를 생성

을 해야한다.

이 시점 이후에 프레임의 입력이 확정된다.

Engine에서 보는 입력은 이 InputState하나뿐이다.

3. SwapBuffers는 무엇인가

InputState가 프레임의 시작을 고정한다면

SwapBuffers는 프레임의 끝을 확정한다.

즉, 이번 프레임 결과를 화면에 보이겠다는 의미다.

다시 말해, 프레임 경계를 OS/Windowing System에 통보하는 행위이다.

그렇지만 렌더링은 RenderBackend에서 한다.

Platform과 RenderBackend는 자주 혼동되지만 이 둘은 책임 축이 완전히 다르다

Platform은 어디를 보여줄지 안다면, RenderBackend는 무엇을 어떻게 그릴지를 안다.

근데 왜 SwapBuffer는 Platform에 있을까?

  • Platform : 윈도우 핸들(HWND)과 그래픽 컨텍스트를 소유함
  • RenderBackend : 그 컨텍스트 위에 이미지를 그림
  • SwapBuffers : OS에게 “이제 이 윈도우에 완성된 그림을 표시해라”라고 명령하는 행위이므로 플랫폼 인터페이스에 위치하는 것이 구조적으로 옳다

SwapBuffers는 Vsync off일 경우 GPU 명령 완료 여부는 신경 안써도 된다.

근데 Vsync를 켜면 SwapBuffers에서 블록 될 수 있다 ( 프레임 페이싱/입력 지연 영향)

대부분의 백엔드에서 Present는 블록될 수 있으며, 엔진은 이를 Render thread로 격리하거나 프레임 큐로 흡수해야 한다

  • Vulkan/D3D12/Metal은 Present가 큐/펜스/세마포어와 강하게 결합되어 있다.
  • OpenGL도 내부 동기화를 걸면 신경써야한다.

4. 정리를 해보자면

엔진루프 관점에선 (메인루프 설계)

Platform 관점에서 필요한 기능을 정리해보면

  1. 한 프레임이 시작할 때, OS 이벤트를 가져와야 함 → PollEvents()
    • OS 메시지 큐 펌프
    • 입력 이벤트 수집
    • 창 이벤트(리사이즈, 포커스 등)
  2. 한 프레임이 끝날 때, 지금까지 그린 것을 화면에 보여줘야함 → SwapBuffers()
    • Double / Triple buffer 교환
    • GPU 명령 완료 여부는 신경 안 써도 됨
  3. OS 상태 체크 → ShouldClose()
  4. 타겟의 실제 픽셀 크기 → GetFrameBufferSize()
    • DPI 대응
    • Retina / HiDPI
    • 실제 GL Viewport 크기
  5. 논리 크기 → GetWindowSize()
  6. 상태, 프레임 스냅샷 읽기 → GetInputState()
  7. OS / 하드웨어 시간 → GetRawTimeSeconds()

둘을 연결하면

5. 이 구조로 Headless 엔진 만들기

그래픽 카드가 없는 서버환경이나 유닛 테스트 환경을 위해 HeadlessPlatform을 준비해야한다.

이렇게 하면 시뮬레이션 엔진이 된다.

6. 이 구조로 게임 만들기

이 구조를 게임에도 사용할 수 있다.

게임과 시뮬레이션 모두 엔진이 하는 일은 동일하다.

이 흐름은 동일하지만, 차이는

게임은 보여주는 방식이 훨씬 훨씬 중요하다!

지금 구조를 다시 보면

  • Platform
    • 입력 수집
    • 시간 제공
    • 화면 제출
  • Engine
    • fixed timestep 시뮬레이션
    • 상태 관리
  • RenderBackend
    • 그래픽 API를 이용한 그리기

여기서 게임다운 요소는 전부 RenderBackend + 상위 시스템에 있다.

  • 애니메이션
  • 카메라 보간
  • 파티클
  • UI
  • 이펙트
  • 사운드 트리거

게임을 만들때는 다음 요소들을 추가하면 된다.

Engine 상위 계층

  • GameWorld
  • Entity / Component
  • Gameplay System
  • State Machine

RenderBackend / Render Layer

  • Sprite / Mesh Renderer
  • Camera System
  • Animation Blending
  • UI Renderer

이러한 입력 구조는 게임에 더 유리하다

게임에서 가장 흔한 버그 중 하나는 이것이다.

  • 어떤 시스템은 키를 눌렀다고 인식하고
  • 어떤 시스템은 이미 뗐다고 인식함

이건 입력 시점이 통제되지 않았기 때문이다.

지금 구조에서는:

  • InputState는 프레임 시작에 한 번 고정
  • 모든 시스템이 동일한 입력 스냅샷 사용

이 한 줄로:

  • 플레이어 이동
  • 카메라 회전
  • UI 입력
  • 디버그 입력

이 전부가 동일한 입력을 기준으로 동작한다.

이건 게임에서 굉장히 큰 안정성을 준다.

그리고 Fixed Timestep + Interpolation 구조는 게임에서도 더 안전하다.

Interpolation으로 시각적 표현을 부드럽게 해줬기 때문이다.

또한 현재 구조는 입력 스냅샷이 명확하고 시간 흐름이 결정론 적이고 플랫폼 의존성이 제거되어있다.

그래서 자연스럽게 싱글플레이/멀티플레이/리플레이 구조로 이어진다.

  • 입력 기록 → 리플레이
  • 입력 동기화 → 멀티플레이
  • Headless 서버 → authoritative server
  • 클라이언트 → 시각화 전용

따라서 이 구조는 로컬게임→온라인게임 으로 확장 가능한 구조이다.

8. 결론 및 요약

플랫폼 설계의 핵심은 엔진을 외부 환경으로부터 고립시키는 것이다

  1. 엔진이 루프를 소유하여 실행 흐름을 통제
  2. 입력 데이터는 누적 및 스냅샷 방식을 사용하여 프레임 내 일관성을 유지
  3. Headless를 고려한 추상화를 통해 서버와 클라이언트가 동일한 로직 코드를 공유하게 하기

이러한 구조는 추후 리플레이 시스템, 네트워크 동기화, 멀티 플랫폼 이식 시 발생할 수 있는 수많은 예외 상황을 사전에 차단해 준다

[부록]이전에 쓴 게임 루프 Game Loop 글을 Platform 관점에서 이해해보자

  1. GetRawTimeSeconds()

    main Loop에 GetRawTimeSeconds()라는 함수가 있다.

    Platform 관점에서 Fixed Timestep Loop를 다시 보면 이건 platform에서 제공하는 것이다.

    이 값은 그냥 OS 기준 단조 증가 시계값이다.

    frameTime은 Platform에서 보내지만 엔진에선 신뢰할 수는 없다.

    그래서 clamp를 추가한다.

    이 클램프는 단순 방어코드가 아니다.

    • OS 스케줄링
    • 디버거 중단
    • 창 포커스 이동

    이 모든 것은 Platform 레벨의 문제다.

    즉, Platform은 시간을 보내고, Engine은 그 시간을 신뢰할지 판단한다.

  2. 입력은 fixed update에 종속되지 않는다

InputState는 프레임 시작에 한번, 물리 업데이트는 여러번 이루어진다.

즉, 하나의 InputState로 여러번의 물리 업데이트가 실행될 수 있다.

  1. interpolation은 렌더링 문제이지 시뮬레이션 문제가 아니다.

    시뮬레이션은 항상 고정된 dt로 진행된다.

    interpolation은 렌더링을 부드럽게 보이기 위한 후처리이다.

    즉, 논리 업데이트 횟수가 변하는 것이 아니라 렌더링 시각화 보정만 하는것이다.

    이건 Platform의 SwapBuffers() 개념과도 정확하게 대응된다.

    • InputState → 프레임 시작 고정
    • SwapBuffers → 프레임 끝 확정
